## 개요

claude-devkit의 에이전트들은 소프트웨어 개발의 각 단계를 전문적으로 담당합니다. 각 에이전트는 명확한 책임 범위를 가지며, 다른 에이전트의 영역을 침범하지 않습니다.

### 설계 원칙

- **단일 책임**: 각 에이전트는 하나의 역할만 수행
- **명확한 경계**: 에이전트 간 책임이 겹치지 않음
- **협업 가능**: orchestrator 스킬을 통해 에이전트들이 순차적으로 협력

## 에이전트 목록

### architect

**소프트웨어 설계 결정 에이전트**

설계 결정만 내리고, 코드를 직접 작성하지 않습니다.

| 속성 | 값 |
|------|-----|
| 모델 | opus |
| 도구 | Read, Grep, Glob |
| 트리거 | "설계해줘", "아키텍처 결정", "인터페이스 정의" |

#### 담당 영역

- **경계 설계**: 모듈/패키지 경계, 바운디드 컨텍스트
- **인터페이스 설계**: API 계약, 메서드 시그니처, 의존성 방향
- **데이터 설계**: 엔티티/VO/DTO 구분, 데이터 흐름
- **트랜잭션 설계**: 트랜잭션 경계, 일관성 보장 전략
- **에러 처리 설계**: 예외 계층, 에러 전파 전략

#### 출력 형식

```
## [설계 영역]: [대상]

### 결정 사항
- [명확한 설계 결정]

### 근거
- [왜 이 결정을 내렸는지]

### 트레이드오프
- 장점: [이점]
- 단점: [비용/제약]

### 계약 (Contract)
- 입력: [기대하는 입력]
- 출력: [보장하는 출력]
```

### implementer

**코드 구현 에이전트**

설계 결정을 바탕으로 실제 동작하는 코드를 작성합니다.

| 속성 | 값 |
|------|-----|
| 모델 | sonnet |
| 도구 | Read, Grep, Glob, Write, Edit, Bash |
| 트리거 | "구현해줘", "코드 작성해줘", "개발해줘", "만들어줘" |

#### 핵심 원칙

- architect의 설계 결정 존중
- 테스트 코드는 qa-engineer가 담당
- 요청받은 기능만 구현 (스코프 확장 금지)
- 과잉 엔지니어링 금지

#### 워크플로우

1. 프로젝트 규칙 파악 (CLAUDE.md, 기존 패턴)
2. 컨텍스트 분석
3. 구현 계획 수립
4. 코드 구현
5. 빌드 검증

### planner

**작업 계획 에이전트**

요구사항을 분석하여 실행 가능한 태스크로 분해합니다.

| 속성 | 값 |
|------|-----|
| 모델 | sonnet |
| 도구 | Read, Grep, Glob |
| 트리거 | "계획 세워줘", "태스크 분해해줘", "우선순위 정해줘" |

#### 담당 업무

- 작업 식별 및 검증
- 태스크 분해 및 의존성 파악
- 우선순위 결정
- TODO 리스트 작성
- 담당 에이전트 할당

#### 출력 형식

```
## 작업 계획

### 작업
1. [T1] [작업명] - [목표]

### 태스크 목록
| 순서 | 태스크 | 담당 | 의존성 | 완료 조건 |
|------|--------|------|--------|-----------|
| 1 | [태스크명] | @architect | - | [조건] |

### TODO 리스트
- [ ] 태스크 1: [설명] (@에이전트명)
```

### qa-engineer

**QA 엔지니어 에이전트**

테스트 케이스를 설계하고 테스트 코드를 작성합니다.

| 속성 | 값 |
|------|-----|
| 모델 | sonnet |
| 도구 | Read, Grep, Glob, Write, Edit, Bash |
| 트리거 | "테스트 작성해줘", "테스트 케이스 설계", "경계값 분석" |

#### 핵심 원칙

- **프로덕션 코드 수정 절대 금지**
- 테스트 코드만 작성
- 테스트 실패 시 이슈로 보고 (직접 수정 안 함)

#### 테스트 설계 영역

- **시나리오 분석**: 정상 흐름, 예외 흐름, 엣지 케이스
- **경계값 분석**: 최소/최대값, 경계 직전/직후
- **동등 분할**: 유효/무효 입력 클래스

#### 출력 형식

```
## [대상] 테스트 케이스

| ID | 구분 | 시나리오 | 입력 | 기대 결과 |
|----|------|----------|------|-----------|
| TC01 | 정상 | [시나리오] | [입력값] | [기대결과] |
| TC02 | 경계 | [시나리오] | [입력값] | [기대결과] |
```

### doc-writer

**문서 작성 에이전트**

프로젝트 문서를 분석하고 작성합니다.

| 속성 | 값 |
|------|-----|
| 모델 | sonnet |
| 도구 | Read, Glob, Grep, Write, Edit, Bash, Task |
| 트리거 | "문서 작성해줘", "README 만들어줘", "가이드 작성해줘" |

#### 문서 유형

- **기술 문서**: README, API 문서, 아키텍처 설계서
- **사용자 가이드**: 튜토리얼, 사용법, FAQ
- **프로젝트 문서**: CHANGELOG, CONTRIBUTING, 릴리스 노트

#### 서브 에이전트 활용

복잡한 정보 수집 시 다른 에이전트 호출:
- **code-explore**: 코드 구조 분석
- **web-explore**: 외부 문서 참조

---

### code-explore

**코드베이스 탐색 에이전트**

코드 구조를 분석하고 구현 위치를 찾습니다.

| 속성 | 값 |
|------|-----|
| 모델 | sonnet |
| 도구 | Read, Grep, Glob, Bash |
| 트리거 | "X는 어디에 구현되어 있어?", "코드 찾아줘", "구조 분석해줘" |

#### 활용 사례

- "인증 로직은 어디에 있어?"
- "이 API 엔드포인트는 어떻게 동작해?"
- "클래스 간 의존성 관계를 알려줘"

---

### web-explore

**웹 탐색 및 요약 에이전트**

웹 검색과 페이지 분석을 수행합니다.

| 속성 | 값 |
|------|-----|
| 모델 | sonnet |
| 도구 | WebFetch, WebSearch |
| 트리거 | "검색해줘", "찾아줘", "~에 대해 알아봐줘" |

#### 활용 사례

- "Spring Boot 4.0 마이그레이션 가이드 찾아줘"
- "이 라이브러리 공식 문서 요약해줘"
- "최신 버전 정보 확인해줘"

## 에이전트 협업 패턴

### TDD 개발 루프 (orchestrator 스킬)

```
사용자 요청
    ↓
[planner] 작업 분해 및 계획
    ↓
[architect] 설계 결정
    ↓
[qa-engineer] 테스트 먼저 작성
    ↓
[implementer] 테스트 통과하는 코드 구현
    ↓
빌드/테스트 검증
    ↓
완료 또는 다음 태스크
```

### 문서화 작업

```
사용자 요청
    ↓
[doc-writer] 문서 유형 파악
    ↓
[code-explore] 코드 구조 분석 (필요시)
    ↓
[web-explore] 외부 참조 (필요시)
    ↓
[doc-writer] 문서 작성
```
